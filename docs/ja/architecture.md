# Sudachi Lookup: 製品とアーキテクチャの仕様

## 1.目的

Sudachi Lookup を使用すると、サーバー側の検索サービスなしで Sudachi 日本語辞典を検索できるようになります。静的アプリケーションと関連するデータ シャードがダウンロードされると、クエリはブラウザー内に残ります。

特徴的なインタラクションは、入力中の即時検索です。 Sudachi C ユニットである結果は、次のように展開して B および A セグメンテーションを表示できます。

```text
A  選挙 / 管理 / 委員 / 会
B  選挙 / 管理 / 委員会
C  選挙管理委員会
```

これは辞書ブラウザであり、当初は任意の文のための汎用形態素解析器ではありませんでした。完全なトークン化は後で検討できますが、最初のデータ形式や配信モデルが複雑になるべきではありません。

## 2.目標

- 検索とランキングをすべてブラウザーで実行します。
- 通常の入力と日本語 IME の使用に従うのに十分な速さで結果を更新します。
- 検索面、辞書、正規化された形式、および読み取り形式。
- ひらがなまたはカタカナで入力された読みを照合します。
- 同形異義語やその他の個別の Sudachi 辞書レコードを保存します。
- Sudachi 独自の分割メタデータを使用して A、B、C セグメンテーションを表示します。
- Cloudflare Pagesへ不変の静的アセットとしてデプロイします。
- 固定されたアップストリーム リリースから辞書の生成を再現できるようにします。
- 初期転送、増分転送、およびブラウザのメモリを制限しておきます。

## 3.最初のリリースの非目標

- 任意の部分文字列、タイプミス耐性のある検索、またはセマンティック検索
- サーバー側 API、認証、アカウント、または同期された履歴
- ブラウザでのユーザー辞書の編集または保守
- 完全な Sudachi トークナイザーを JavaScript で再実装する
- 最初の展開時にすべての SudachiDict エディションを出荷
- 表面テキストを一意の辞書識別子として扱う

## 4.重要な決定

### 4.1 ルックアップ固有のデータセットを生成する

デプロイされたデータは、元のコンパイル済み辞書ではなく、固定された SudachiDict リリースのオフライン派生データになります。ルックアップ製品には、ソートされた検索キーとエントリの詳細へのランダム アクセスが必要です。トークナイザーの接続マトリックスやすべてのランタイム構造は必要ありません。

この分離により、未完成のブラウザと WASM のパスに依存することも回避されます。 2026 年 7 月の時点で、ブラウザーと JavaScript 環境のサポートは、sudachi.rs プロジェクトで未解決のリクエストのままです: [sudachi.rs 問題リスト](https://github.com/WorksApplications/sudachi.rs/issues)。

### 4.2 JavaScript や JSON レコードではなく、コンパクトなバイナリ シャードを使用する

大きな JavaScript モジュールを使用すると、ブラウザーはソース コードを解析し、大きなオブジェクト グラフを割り当てる必要があります。 JSON にも同様の割り当てコストがあり、フィールド名と文字列が繰り返されます。実稼働フォーマットでは、代わりに、測定結果に利点がある `ArrayBuffer`、型付き配列、整数オフセット、共有文字列テーブル、および可変長整数を使用する必要があります。

最初のプロトタイプは、正確性を確立するために MessagePack または CBOR を使用する場合があります。形式を安定させる前に、その出力を小さなカスタム バイナリ エンコーディングと比較します。カスタム形式は、バイト数、デコード時間、またはメモリを大幅に削減する場合にのみ正当化されます。

### 4.3 プレフィックス クエリを制限された静的ファイルにルーティングする

単一のデータベースまたはモノリシック インデックスでは、初回読み込みのパフォーマンスがコーパスの大部分またはすべてのダウンロードに依存します。代わりに、検索アセットは正規化された先頭文字によって分割され、圧縮された各シャードが設定されたサイズの上限を下回るまで細分化されます。

ルーティング スキームはプレフィックスを保持する必要があります。検索キーをハッシュシャーディングすると、プレフィックスクエリがすべてのシャードに分散されるため、適切ではありません。

### 4.4 Sudachi の単語のアイデンティティを保持し、ブラウザの分割境界を解決します

Sudachi には、同じサーフェスを持つ複数のエントリを含めることができます。したがって、抽出された各辞書レコードは、固定された入力から派生した内部安定 ID を受け取りますが、UI レベルのグループ化はオプションのままです。

Sudachi WordInfo は、構造体と A ユニットと B ユニットの分割データを公開します。抽出段階では、検証のためにこれらの関係が単語 ID として保存されます。ブラウザ形式 v4 は、それらを解決して親サーフェス内の累積コードポイント境界を圧縮し、Sudachi の作成されたセグメンテーションを保持しながらコンポーネント レコードのリクエストを回避します。エンコーディングと測定された不変量は、[Compact Split Boundary Format](split-boundary-format.md) に文書化されています。利用可能な上流フィールドは、[SudachiPy WordInfo サブセット](https://worksapplications.github.io/sudachi.rs/python/topics/subsetting.html) に記載されています。

### 4.5 最初に SudachiDict Core を使用する

コアは、最大のエディションから始めなくても、検索品質とストレージの動作を検証するのに十分な範囲を提供する必要があります。 Small と Full は、後で同じパイプラインから生成し、展開の選択肢またはオプションのデータ パックとして提供できます。

その後の完全実現可能性実行では、同じ v3 パイプラインが 4,293 個の完全辞書ファイルを生成し、すべてのファイルが 1 MiB 未満の raw 状態に保たれることが示されました。コアとフルを合わせても、依然として Cloudflare Pages の無料プランのファイル許容量を下回っています。したがって、Pages は両方のエディションのデフォルトのホストです。オブジェクト ストレージは偶発的なものであり、前提条件ではありません。

### 4.6 トークナイザーの語形変化レコードではなく、正規の見出し語を公開する

ニュートラル エクスポートは、固定された Sudachi 辞書のロスレス表現のままですが、ブラウザ データ フォーマット v5 は、トークン化中に使用されるすべての活用状態レコードではなく、正規の辞書形式の ID を公開します。フィルタリングでは上流の辞書形式の単語 ID が使用され、異なる同形異義語が保存されます。表面文字列のグループ化が不十分です。ブラウザー レコード シャードはステージ 1 ではロスレスのままであるため、レコードの圧縮を個別に評価できます。段階的ポリシーとその検証要件は、[正規見出しフィルタリング](canonical-headword-filtering.md) で定義されています。

## 5.論理データモデル

次の TypeScript では、ディスク上のエンコーディングではなく、セマンティクスについて説明します。

```ts
type EntryId = number

interface Entry {
  id: EntryId
  sourceWordId: number
  canonicalSourceWordId: number
  surface: string
  normalizedForm: string
  dictionaryForm: string
  readingForm: string
  posId: number
  structureBoundaries: number[]
  aBoundaries: number[]
  bBoundaries: number[]
  synonymGroupIds?: number[]
  rankingSignal?: number
}

interface SearchAlias {
  key: string
  entryId: EntryId
  kind: "surface" | "dictionary" | "normalized" | "reading"
}
```

対応するセグメンテーションが存在しない場合、境界配列は空になります。各値は、親サーフェス内の累積 Unicode コードポイント オフセットです。オフライン ジェネレーターは、1 バイトのブラウザ境界を出力する前に、元の参照単語 ID とコンポーネントの長さを検証します。コンポーネント ラベルは親サーフェスから再構築され、参照レコードをフェッチせずに文脈上の大文字と正書法を維持します。

形式 v9 および v10 では、すべてのブラウザー レコードに Sudachi の元の `u16` POS ID が保存されます。 1 つの gzip 圧縮された `pos.bin.gz` テーブルは、使用される ID を完全な結合コンポーネント文字列にマップします。ワーカーはブートストラップとともにそれをロードし、レコード ID を共有 JavaScript 文字列に解決するため、UI コントラクトは変更されません。空のフォームまたは同一のフォームは、今後の形式でも重複文字列ではなく、`surface` への参照として表現される可能性があります。

画面に表示するSmall・Core・Fullの収録区分も、各レコードには保存せず導出します。固定したリリースでは、Fullの語彙はSmallと同一のword ID接頭範囲、その後にCoreの追加分、さらにFullの追加分という順序です。このため、マニフェストに二つの境界値を持つだけで、正規見出し語を収録する最小のエディションを判定できます。この不変条件とリリース時の検証は[辞書エディションの収録区分](edition-membership.md)で定義します。

## 6.クエリの正規化

インデックス生成時に、有用な Sudachi フォームのエイリアスを発行します。クエリ時:

1. Unicode NFKC 正規化を適用します。
2. インデックスジェネレーターと一貫して小文字のラテン語テキスト。
3. 表面検索用に正規化されたリテラル クエリを保持します。
4. 読み方検索用にカタカナとひらがなの異体字を生成します。
5. ユーザーテストで必要性が示されない限り、最初のリリースではローマ字からの音訳を避けてください。

Sudachi は、NFKC ベースの入力正規化を実行し、辞書固有の正規化形式を保存します。提供された正規化された形式にインデックスを付ける必要があります。 NFKC のみを適用することは Sudachi 正規化と同等ではありません。 [Sudachi プロジェクト ドキュメント](https://github.com/WorksApplications/Sudachi#normalized-form) を参照してください。

IME 構成イベントは明示的に処理する必要があります。インターフェイスは、動作が有用であることがテストで示されない限り、不安定な中間構成テキストに基づいて結果を継続的に置き換えるべきではありません。

## 7.検索インデックス

### 7.1 2 層ルックアップ

日本語の接頭辞の中には、複数の文字の後であっても、辞書の非常に大きな部分を表すものがあります。完全な投稿をロードすると、キーストロークが不釣り合いに遅くなります。

したがって、アプリケーションは 2 つの層を使用します。

1. **ブートストラップの提案:** 事前に計算された上位 20 件の結果と、完全な検索が高価なプレフィックスの表示レコードを含む、熱心に読み込まれる小さなインデックス。
2. **完全なプレフィックス シャード:** 合理的に制限されたデータ パーティションにルーティングするのに十分な期間、またはユーザーが広範なクエリのブートストラップ結果を超えて続行する場合に、クエリの並べ替えられたエイリアスを遅延読み込みします。

フォーマット v7 では、クエリの長​​さではなく生成されたコスト統計からのブートストラップ選択が導入されており、フォーマット v8 から v10 ではそのモデルが維持されています。プレフィックスは、少なくとも 500 のエイリアスと一致し、少なくとも 192 KiB の検索データをルーティングする場合、またはその最初の結果に少なくとも 1 MiB のレコード シャードが必要な場合に適格となります。ジェネレーターは、一致するエイリアス 100 個までのブランチを探索するため、`あきの` などのレコード分散クエリも対象となります。適格なプレフィックスは、回避される検索シャード転送とレコードシャード転送の組み合わせによって優先順位が付けられ、現在のハード デコードされた 2.5 MiB バジェットの下でエンコードされます。複数のプレフィックスによって共有されるレコードは 1 回だけ保存されます。ブートストラップ自体は転送用に gzip 圧縮され、ワー​​カー内で解凍されます。ひらがなとカタカナの綴りには、異なるブートストラップ キーとランキングがあります。ブートストラップとライブ シャード ランキングは両方とも、リテラル クエリ スクリプトを正確性とエイリアスの種類の後のタイブレーカーとして扱うため、`あま` はそれ以外は等しいひらがなサーフェスを優先し、一方、`アマ` はカタカナを優先します。継続は依然として完全なプレフィックス シャードに拡張されます。

### 7.2 シャード ルーティング

マニフェストは、クエリ プレフィックスを 1 つ以上のファイルにマップします。概念的なレイアウトは次のとおりです。

```text
public/data/
  manifest.json
  20260428-core-v1/
    bootstrap.ab12cd.bin
    pos.0237a1.bin
    search/
      せん.40f1be.bin
      選挙.249a91.bin
    records/
      000.92dba2.bin
      001.88ea31.bin
    notices/
      NOTICE.txt
      LICENSE-2.0.txt
```

最終的な実装では、論理プレフィックスは移植可能な URL セーフなファイル名または不透明な ID に変換する必要があります。上記の人が判読できる名前は例示です。

ビルド プロセスは先頭のパーティションから始まり、その圧縮サイズを測定し、大きなパーティションを再帰的に細分化します。ターゲットは、プロトタイプの結果に応じて、クエリ シャードごとに圧縮された 100 ～ 300 KiB などの制限された転送です。

### 7.3 投稿内容

検索投稿には、すべてのエントリの詳細を取得しなくても、最初の結果フレームをランク付けして表示するのに十分な情報が含まれている必要があります。候補フィールドには次のものが含まれます。

- エントリーID
- 表示面基準
- 参考文献を読む
- POS ID
- 別名の種類
- ランキングシグナル

完全に正規化された辞書形式、同義語グループ、分割参照は、結果が選択または展開された後にロードされたレコード シャードに残ることができます。

### 7.4 ランキング

最初の決定論的な順序付けでは、以下を優先する必要があります。

1. 表面の正確な一致
2. 辞書形式または正規化形式の完全一致
3. 表面接頭辞一致
4. 読み取り値が完全に一致
5. 前方一致の読み取り
6. 一致強度が等しい場合のリテラルのひらがなまたはカタカナ スクリプト
7. 短い表面形状
8. 抽出と解釈が検証された場合、信頼できる辞書コストまたは周波数のような信号
9. 最終的なタイブレーカーとしての安定したエントリー ID

同じエントリを指すエイリアスは、レンダリング前にマージされます。表面は同じだが言語記録が異なるエントリを黙って失ってはなりません。

## 8.ブラウザのランタイム

検索とデコードは Web Worker で実行されるため、入力、IME の作成、レンダリングの応答性が維持されます。

```text
Input event
    |
    v
Query normalization and monotonically increasing request ID
    |
    v
Web Worker -> manifest route -> shard cache/fetch -> binary search -> ranking
    |
    v
Stable top-result IDs returned to UI -> placeholder cards
    |
    v
Cached records rendered immediately
    |
    v
Record shards settle independently -> matching cards filled in place
```

ランタイムは次のことを行う必要があります。

- 軽くデバウンスします (テストの対象) (約 40 ～ 80 ミリ秒)。
- 古いリクエスト ID からの結果を破棄します。
- 実際的な場合は、正確性をアボートに依存せずにフェッチをアボートします。
- デコードされたシャードをメモリ境界の LRU キャッシュに保持します。
- ブラウザーの HTTP キャッシュが圧縮された応答を保持できるようにします。
- 空ではないクエリが開始されるとすぐに、約 20 枚のプレースホルダー カードを表示します。
- 結果 ID を公開する前に検索シャードのランキングを完了することで、暫定エントリが間違った順序で表示されることがなくなります。
- これらの ID を安定したスロットにロックし、ブートストラップまたはキャッシュされたレコードをすぐにレンダリングし、各レコードのシャードが落ち着いたら残りのスロットを埋めます。
- リクエスト ID が古いすべてのスロットまたはレコードの更新を無視します。
- 最初に約 20 件の結果を返し、ユーザーがリストの最後に近づくと継続ページを追加します。
- 広範なブートストラップ検索が完全なプレフィックス シャードに拡張される場合、すでに表示されているエントリと安定した重複排除が維持されます。
- 新しいクエリがアクティブになったときに、継続作業をキャンセルまたは無視します。
- キーボード ナビゲーションとアクセス可能な結果アナウンスをサポートします。

Service Worker はオプションです。アプリケーション シェルとブートストラップ インデックスをキャッシュする場合がありますが、最初のリリースでは通常の不変 HTTP キャッシュで十分です。

## 9.パイプラインの構築

辞書ビルダーはバージョン管理されたコマンドライン プログラムであり、ブラウザーで実行されることはありません。

```text
Pinned edition and release
    -> verify source checksum
    -> extract lexical records and upstream IDs
    -> resolve A/B split information
    -> normalize and generate search aliases
    -> intern repeated strings and POS values
    -> build and size-balance prefix shards
    -> measure prefix cost and build budgeted bootstrap results
    -> build record shards
    -> emit manifest, hashes, statistics, and notices
    -> run integrity and search fixtures
```

選択した Sudachi リリースに固定された小さな Rust エクスポータをアップストリーム アダプタとして使用します。 SudachiPy は正確な表面検索を提供しますが、完全なレキシコンの反復は提供しません。一方、Rust レキシコンはそのサイズと単語情報を ID によって公開します。エクスポーターは、公式にコンパイルされた辞書を使用し、Web フォーマット ビルダー用の中立的なエントリ ストリームを生成します。正確な上流コミットを固定し、プライベート バイナリ レイアウトを個別に解析することを避ける必要があります。

マニフェストには少なくとも以下が含まれている必要があります。

  "generatorVersion": "0.1.0",

  "posTableFile": "pos.bin.gz",
  "posCount": 1558,
  "posTableBytes": 10587,
  "posTableDecodedBytes": 113109,
  "posCompression": "gzip",
  "posEncoding": "sudachi-u16",

  "bootstrapPrefixes": 4071,
  "bootstrapRecords": 39963,

### 9.1 ディクショナリ リリース アーティファクト

辞書の生成とサイトの展開は別のパイプラインです。辞書リリース パイプラインは、固定された上流のエディションとバージョンを使用し、すべてのデータ検証を実行し、マニフェスト、統計、チェックサム、通知を含むコンテンツ アドレス指定されたアーティファクトを公開します。アーティファクトは、上流のディクショナリ バージョンまたはジェネレータ バージョンが変更されるまで再利用されます。

通常のフロントエンド ビルドでは SudachiDict を再生成してはなりません。ローカル開発とプルリクエストのプレビューでは、小さな決定論的なフィクスチャが使用されます。ステージングと実稼働では、すでに検証済みのコアまたはフル アーティファクトを選択します。

フルの場合、推奨される運用パスは CI 所有のアセンブリとデプロイメントです。

```text
Dictionary release workflow
    SudachiDict Full -> generate -> validate -> publish versioned artifact

Site deployment workflow
    frontend + selected artifact -> build dist/ -> validate -> Wrangler upload
```

これにより、高価なフルビルドがローカルフィードバックループとCloudflare Pagesのビルド環境の両方から排除されます。また、デプロイされたものを正確に表す 1 つの検査可能な `dist/` アーティファクトも生成されます。

## 10.検証要件

### データの整合性

- 参照されたすべての A/B 分割コンポーネントが解決されます。
- すべての検索投稿はレコードに解決されます。
- 分割結果は、フィクスチャの固定された Sudachi 実装と一致します。
- 重複したサーフェスは表示されたままになります。
- UTF-8 と文字列テーブルのオフセットは範囲内に留まります。
- マニフェスト ハッシュは発行されたファイルと一致します。
- 同じピン留めされた入力からのビルドは、実用的な場合にはバイトごとに再現可能です。

### 検索動作

- 正確な曲面、正規化形式、辞書形式、および読み取り治具が合格します。
- ひらがなとカタカナのクエリは、同等の読みの一致を返します。
- ランキングは決定的です。
- 古いワーカーの応答が新しい結果に置き換わることはありません。
- 空、句読点のみ、ラテン語、数値、および非常に長いクエリは安全です。

### パフォーマンス予算

プロトタイプは推測ではなくレポートする必要があります。

- 圧縮出力と非圧縮出力の合計
- ブートストラップの転送時間とデコード時間
- 中央値、p95、および最大シャード サイズ
- コールドクエリとウォームクエリのレイテンシ
- 代表的なセッション中のワーカーのピークメモリ
- 代表的なクエリシーケンスに対するネットワークリクエストの数
- A/B 分割データを含むエントリの範囲

具体的なリリース予算は、最初のコア ビルド後に確立する必要があります。

## 11.ローカル開発とCloudflare Pagesの展開

### 11.1 ローカル ワークフロー

最初のブラウザ プロトタイプのローカル HTTP 開発サーバーを使用します。モジュール ワーカー、`fetch`、MIME 処理、およびキャッシュには現実的な HTTP 動作が必要であるため、`file://` を介してアプリケーションを開くことはサポートされているワークフローではありません。

対象となるコマンドは概念的には次のとおりです。

```text
npm run data:sample   Generate the small deterministic fixture
npm run dev           Run the local development server
npm test              Run data and search tests
npm run build         Build static production output
npm run preview       Serve the production output locally
```

完全な辞書の生成は明示的な解放操作であり、`npm run dev` の前提条件ではありません。

### 11.2 ページ プロジェクトのタイミング

最初の垂直スライスが機能した後、Cloudflare Pages プロジェクトを作成します。本番フロントエンドビルドは、ワーカーを通じてサンプルシャードをロードし、実際の結果を返し、少なくとも 1 つの A/B/C エントリを展開する必要があります。その時点でページを作成すると、ホスティング設定がデータ プロトタイプをブロックすることなく、実際の圧縮、キャッシュ、MIME タイプ、ファイル制限、ネットワーク遅延がテストされます。

開発中に `pages.dev` デプロイメントと Wrangler によって作成されたブランチ プレビュー URL を使用します。選択した実稼働セットアップは Pages Direct Upload プロジェクトであるため、CI はアップロード前に 1 つの辞書エディションだけを組み立てて検証できます。直接アップロードでは、Git によってトリガーされるネイティブのプレビューは提供されず、後で Git 統合に変換することはできません。この個人プロジェクトでは明示的なプレビュー展開が許容され、フル リリース パイプラインの決定性が維持されます。

完全なリリース候補がデータの整合性、ライセンス、キャッシュ、アクセシビリティ、およびモバイル パフォーマンスのチェックに合格した後にのみ、運用カスタム ドメインに接続してください。オプションのステージング サブドメインは、安定したプレビュー ブランチを指す場合があります。

### 11.3 本番配信

実稼働サイトは、静的なアプリケーション ファイルと辞書アセットのみで構成されます。 Pages Function、D1、KV、R2 は初期設計では不要です。

現在の Vinext ビルドは、最終的な Pages アーティファクトではなく、中間アプリケーション ビルドです。スタンドアロンの静的 HTML エントリ ポイントを持たずに、Worker エントリ ポイントとブラウザ アセットを生成します。ページ固有のアセンブリ ターゲットは、クリーンな静的ディレクトリを作成する必要があり、明示的な `sample`、`core`、または `full` の選択を必要とします。作業ツリーにたまたま存在する無視されたデータから、デプロイされたエディションを決して推測してはなりません。

フル エディションの場合、CI は実稼働ビルドを所有します。検証された辞書アーティファクトを取得し、フロントエンドを構築し、完全な `dist/pages/` ディレクトリをアセンブルしてテストし、Wrangler を使用して Pages にアップロードします。 Pages は、高価な辞書の構築が行われる場所ではなく、静的なホストです。

最初のアップロードでは、プレビュー ブランチのサンプル フィクスチャを使用します。コアは、キャッシュとコンテンツ タイプの動作が証明された後、別のステージング ブランチをたどります。フルは、コアがネットワークとブラウザのパフォーマンス バジェットを確立した後、明示的な CI リリース ジョブにのみ付加されます。

バージョン管理され、コンテンツがハッシュされたデータ ファイルは以下を使用できます。

```text
Cache-Control: public, max-age=31556952, immutable
```

デプロイメントを新しいバージョンのデータ ディレクトリに自動的に切り替えることができるように、HTML と現在のマニフェストは再検証可能な状態を維持する必要があります。 Cloudflare Pagesは、静的アセットのカスタム`_headers`ルールとBrotli配信をサポートしています。

- [ヘッダー](https://developers.cloudflare.com/pages/configuration/headers/)
- [ページの提供](https://developers.cloudflare.com/pages/configuration/serving-pages/)

Cloudflare Pages では現在、仕様に準拠した部分的な `206` 応答ではなく、HTTP 範囲リクエストに対する `200` 応答を文書化しているため、この設計ではリモート SQLite 範囲の読み取りを意図的に回避しています。また、Pages は個々の静的アセットを 25 MiB に制限します。これも、制限されたシャードを生成するもう 1 つの理由です。

- [ページの提供](https://developers.cloudflare.com/pages/configuration/serving-pages/#behavior)
- [ページ制限](https://developers.cloudflare.com/pages/platform/limits/)

カスタム サブドメインは、最も単純な運用セットアップです。 Cloudflareは、サブドメインと頂点ドメインの両方の構成を[カスタムドメイン](https://developers.cloudflare.com/pages/configuration/custom-domains/)に文書化しています。

## 13.リスクと緩和策

|リスク |緩和 |
| --- | --- |
|非常に大きな最初の文字のパーティション |上位の結果と再帰的にサイズ制限されたシャードをブートストラップする |
|ブラウザのメモリが過剰です |ワーカーのデコード、コンパクトなアレイ、LRU バジェット |
|重複または曖昧なエントリ | UI 内でのみ安定したエントリ ID とグループを保持する |
|分割参照はリリース間で変更されます |ビルドごとにリリースを固定し、フィクスチャを検証する |
|バイナリ形式の進化は困難になる |形式をバージョン管理し、マニフェストを明示的に保ちます。
|コーパス頻度がないと検索品質が低下します。決定論的な言語ランキングから始めて、実際のクエリを測定します。
|新しい辞書の展開には古い資産と新しい資産が混在しています |コンテンツハッシュ化された不変ファイルと 1 つのバージョン選択マニフェスト |
|アップストリーム抽出 API の変更 |テスト済みのアダプターとピンツールのバージョンの背後で抽出を分離する |
|ライセンス通知が不完全です |正確に固定されたリリースからのパッケージ通知と公開前のレビュー |
|フルビルドでは通常の開発が遅くなる |チェックインされたサンプル フィクスチャを使用し、完全なものを再利用可能な CI アーティファクトとして公開します。
|展開を再現できません |検証された辞書アーティファクトと正確に組み立てられた `dist/` メタデータを保持します。

## 14.配信ロードマップ

### フェーズ 1: データの実現可能性

- SudachiDict Core リリースを 1 つピン留めします。
- エントリを抽出し、関係を分割します。
- 検索エイリアスとサイズバランスの取れたシャードを生成します。
- 整合性、サイズ、遅延、メモリのレポートを作成します。
- 測定値から初期バイナリエンコーディングを決定します。

### フェーズ 2: 機能的なブラウザのプロトタイプ

- ローカル開発サーバーと確定的なサンプルデータ コマンドを追加します。
- ワーカー プロトコルとシャード キャッシュを実装します。
- インクリメンタル検索と決定論的ランキングを追加します。
- レンダリング エントリ メタデータと A/B/C セグメンテーション。
- IME、キーボード、モバイル、およびアクセシビリティの動作をテストします。

### フェーズ 3: ホストされたプレビュー

- リポジトリを Git ホストにプッシュします。
- 垂直スライスがローカルで機能した後、Cloudflare Pages プロジェクトを作成します。
- `pages.dev` およびブランチ プレビューを通じてサンプル データのビルドを実行します。
- バイナリ MIME タイプ、ヘッダー、圧縮、キャッシュの無効化を検証します。

### フェーズ 4: 実稼働コア サイト

- インターフェイスと応答性の高いプレゼンテーションを完成させます。
- 辞書版とお知らせページを追加しました。
- 検証されたコア辞書アーティファクトを公開して使用します。
- 不変のアセットヘッダーと運用環境のデプロイの自動化を構成します。
- Cloudflare Pagesにデプロイし、カスタムドメインに接続します。

### フェーズ 5: フルエディション

- 固定された完全リリースを生成して測定します。
- CI 所有の完全なアーティファクトとサイト展開ワークフローを追加します。
- Wrangler を使用して完全な静的出力を組み立て、検証し、アップロードします。
- ページのファイル数とアセットごとの制限が適切なマージンを保持していることを確認します。

### フェーズ 6: オプションの拡張

- Small Edition を追加します。
- 明示的なオフライン/フルデータのダウンロードを提供します。
- 観測された需要から部分文字列またはあいまい検索を評価します。
- 語彙検索とは独立して、任意のテキストのトークン化を評価します。
- [構造一致ルックアップ プラン](structure-match-lookup.md) で指定された、遅延した ID ベースの構造逆引きルックアップを評価します。
- オプションで、中立的な Web 検索アクションの背後にあるプロバイダーを構成可能にします。

## 15.未解決の質問

これらは、実現可能性のあるプロトタイプまたは製品テストによって回答される必要があります。

- どの上流抽出パスが最も安定しており、再現可能ですか?
- 同形異義語を保存した結果、コア レコードとエイリアスは何個できますか?
- 一致の種類と長さ以外に適切なランキング シグナルがある場合、どのようなシグナルが適切ですか?
- どのシャードの上限がレイテンシとリクエスト数のバランスを最適化しますか?
- 短いプレフィックスの提案はグローバルにする必要がありますか、それともエイリアスの種類で区切るべきですか?
- コンテキスト表面のテキストは分割エントリ参照を伴う必要がありますか?
- 結果のグループ化は言語エントリまたは表示される見出し語をデフォルトにする必要がありますか?
- Full は運用環境で Core を置き換えるべきですか、それとも選択可能なエディションとして提供されるべきですか?
- 検証された辞書成果物はどこに、どのくらいの期間保持する必要がありますか?

サーバー側コンポーネントなしで、コア プロトタイプが合意された転送、レイテンシ、メモリ、整合性、および検索品質の予算を満たしている場合、アーキテクチャは検証されたとみなされます。
