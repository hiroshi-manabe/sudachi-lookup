# コンパクトな分割境界フォーマット

## 決定

ブラウザー データ形式 v4 は、構造、A、および B セグメンテーションを、親エントリの `surface` 内の累積 Unicode コードポイント境界として保存します。参照された単語 ID はブラウザー レコード シャードに保存されません。フォーマット v5 から v10 は、同じ分割表現を維持しながら、正規の見出し語検索フィルタリングとコスト主導のブートストラップを追加します。

例えば：

```text
surface:   選挙管理委員会
Structure: [2, 4]       -> 選挙 / 管理 / 委員会
A:         [2, 4, 6]    -> 選挙 / 管理 / 委員 / 会
B:         [2, 4]       -> 選挙 / 管理 / 委員会
```

最後のコンポーネントはサーフェスの残りの部分であるため、`n` コンポーネント シーケンスには `n - 1` 境界が必要です。空の境界リストは、引き続き対応するセグメンテーションが存在しないことを意味します。 2 つのモードの境界が同一である場合でも、3 つのフィールドは論理的に区別されたままになります。

各境界は符号なしバイトです。 UTF-8 バイトや JavaScript UTF-16 コード単位ではなく、Unicode コード ポイントをカウントします。ブラウザは、補助文字が誤って分割されないように、`Array.from(surface)` からコンポーネントを再構築します。

## 境界だけで十分な理由

ブラウザは、分割コンポーネントを表示テキストおよび新しい表面クエリとして使用します。参照されるコンポーネントのエントリの読み、品詞、その他の語彙フィールドは使用されません。したがって、親サーフェスからコンポーネント テキストを派生すると、ユーザーに表示されるスペルが保持され、コンポーネント レコード シャードをロードする必要がなくなります。

固定された `20260428` 辞書の測定値は、この表現をサポートしています。

|測定 |コア |フル |
| --- | ---: | ---: |
| A/B 分割を含む最長のサーフェス | 133 コードポイント | 133 コードポイント |
| 127 コード ポイントを超えるサーフェス | 1 | 1 |
| 255 コード ポイントを超えるサーフェス | 0 | 0 |
|コンポーネントの長さは親をカバーします | 100% | 100% |
|構造テキストが正確に連結される | 99.9966% |エンコーディングでは必要ありません |
|テキストは正確に連結されます | 99.9966% |エンコーディングでは必要ありません |
| B テキストは正確に連結されます。 99.9945% |エンコーディングでは必要ありません |

28 のコア テキストの不一致は、親 `Aiロボティクス` と参照コンポーネント `AI` および `ロボティクス` などの大文字化のバリアントです。それらのコンポーネントの長さは依然として有効な境界を定義します。親から再構築すると、意図的に `Ai / ロボティクス` が表示されます。これにより、結果自体のスペルが保持され、正規化された検索と同等のままになります。

コアの現在の A/B ID 配列は約 17.03 MiB を占有します。 1 バイト境界には約 3.94 MiB が必要で、同じ変更を構造に適用する前に約 13.1 MiB 削減されます。

## 生成と検証

オフライン抽出レポートでは、上流の単語 ID が引き続き保持されます。 Web フォーマット ビルダーは、ピン留めされたエントリ サーフェスに対してこれらの ID を解決し、次のすべてを検証した後にのみ境界を出力します。

- 参照されたすべての単語 ID が解決されます。
- 存在する場合、すべての分割には少なくとも 2 つのコンポーネントが含まれます。
- コンポーネントのコードポイントの長さの合計は親サーフェスの長さになります。
- 境界は厳しく増加しています。
- すべての境界は親の長さより短く、256 未満です。

不変条件が成立しない場合、ビルドは、あいまいな分割を黙って発行するのではなく、失敗する必要があります。新しい辞書で 256 以降の境界が導入された場合、形式は v4 バイトの意味を変更する代わりに、明示的なエスケープまたは新しいバージョンを追加する必要があります。

## 読み込み動作

構造、A、および B の境界は、親サーフェスと同じレコードに積極的に保存されます。これらのコンパクトな表現は既存の ID 配列よりも小さいため、遅延分割詳細ファイルは、意味のある転送の節約がなければ、リクエストとキャッシュの複雑さを追加します。

コールド検索は次のようになります。

1. 関連する検索シャードをロードします。
2. 表示されている親結果のレコード シャードを読み込みます。
3. コンポーネント レコード リクエストなしで構造と利用可能なすべての分割モードをレンダリングします。

分割行は要求されるまで視覚的に折りたたまれたままですが、それを開くために追加のネットワーク アクセスは必要ありません。

## 実装結果

どちらの固定エディションも v4 の生成と検証に合格します。検索パーティショニングとエイリアス ペイロードは変更されません。バイナリ ヘッダーとマニフェストには新しい形式のバージョンが含まれます。

|記録データ | 252,802,733 B | 232,461,377 B | 471,957,614 B | 427,577,506 B |
|辞書データ合計 | 489,621,289 B | 469,279,985 B | 913,268,231 B | 868,888,170 B |

この変更により、コアで 20,341,304 バイト、フルで 44,380,061 バイトが節約されると同時に、通常の結果ハイドレーションから 2 回目のレコード読み込みラウンドが削除されます。
