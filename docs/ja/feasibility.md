# 最初の実現可能性の調査結果

## 垂直スライスを実装しました

最初のローカル マイルストーンが実装され、本番環境で計画されているのと同じ境界を通過します。

```text
fixture source
    -> deterministic binary generator
    -> entry and search-index files
    -> HTTP fetch
    -> Web Worker decode and prefix lookup
    -> ranked browser results
    -> A/B/C expansion
```

このフィクスチャには 29 の代表的なエントリが含まれており、145 の検索可能なエイリアスが生成されます。これには、`今日` の両方のコア読み取り値 (`キョウ` および `コンニチ`) が含まれます。 HTTP 圧縮前の 2 つのバイナリ ペイロードは合計 4,820 バイトです。対象範囲には、表層形式、正規化形式、辞書形式、カタカナ形式、およびひらがな形式が含まれます。同形異義語。分割されていないエントリ。 A と B の参照を持つ C エントリ。

現在のバイナリ形式は意図的に小さく、バージョン管理されています。これは実行時の境界を証明しますが、まだ運用シャーディング形式ではありません。

## 固定された上流の実験

最初の実データ調査では以下が使用されました。

- すだちPy `0.6.11`
- SudachiDict コア `20260428`
- Python `3.12`

インストールされているコンパイル済みコア辞書は 217,374,303 バイトで、SHA-256 が含まれています。

```text
6c1d5adc8a2389875713056e7b39bbcd0073d6122ffd509866e1d3a196f8608e
```

このチェックサムは、ローカルで検査されたインストール済みの `system.dic` を表します。リリース ワークフローは、公式入力アーティファクトを個別に検証し、独自のチェックサムを記録する必要があります。

## コア抽出結果

SudachiPy は厳密な表面検索をサポートしていますが、文法や語彙に対する反復を意図的に公開していません。正確な検索ではすべてのサーフェスを検出できないため、完全なブラウザ データセットを生成できません。

サポートされている Rust 構造は、エクスポーターが必要とするプリミティブを提供します。

- `DictionaryLoader::read_system_dictionary`
- `Lexicon::size`
- `Lexicon::get_word_info`
- `Lexicon::get_word_param`

ビルド時の Rust エクスポータは Sudachi `v0.6.11` コミットに固定されています。

```text
90fd6068c80c2fc3b63e0dbab0e341475bad4d8f
```

将来の Web 形式ビルダーのために、gzip 圧縮された JSON Lines ストリームを出力します。このニュートラル ストリームは中間ビルド アーティファクトであり、ブラウザー ペイロードではありません。上流のリーダーを Web 形式ビルダーから分離しておくと、Sudachi の内部 API の変更が分離されます。

Apple Silicon 開発マシンでの次の測定により、完全な Core 辞書が正常にエクスポートされました。

エクスポートされた行数はちょうど 1,629,080 です。予想される 2 つの `今日` エントリが、読み取り値 `キョウ` および `コンニチ` で存在します。代表的なコンパウンド `選挙管理委員会` は、次のようにアップストリーム ID から直接解決されます。

```text
A  選挙 / 管理 / 委員 / 会
B  選挙 / 管理 / 委員会
C  選挙管理委員会
```

エクスポータは、ソースと出力 SHA-256 チェックサムをレポートに記録します。生成されたレポートとニュートラル ストリームは `reports/` の下に存在し、再現可能で比較的大きいため、Git によって無視されます。

次のコマンドを使用して、固定されたコアのエクスポートを実行します。

```sh
npm run data:core
```

ラッパーはデフォルトでプロジェクトローカルの Rust ツールチェーンと Core ディクショナリを使用します。 `SUDACHI_SYSTEM_DIC` および `SUDACHI_RELEASE` は、インストールされている別の辞書をテストするときに、これらのパスとレポート ディレクトリ名をオーバーライドします。

## 次の受付ポイント

列挙、関係の検証、および最初の Web 形式のビルドが完了しました。 v3 プロトタイプは、8,140,​​461 個の重複を排除したサーフェス、辞書、正規化されたカタカナ読み、およびひらがな読みのエイリアスを生成しました。

| --- | ---: | ---: | ---: | ---: | ---: |
|シャードの検索 | 1,629 | 131,315 B | 247,187 B | 400,838 B | 236,221,915 B |

ブートストラップ インデックスには 15,296 個のエイリアスが含まれており、生のサイズは 303,834 バイトです。マニフェストは未加工の 292,806 バイトです。完全に生成されるディレクトリは、HTTP 圧縮前で約 472 MiB ですが、単位としてダウンロードされません。通常のクエリは、重複するソート キー範囲と、表示される結果とその分割コンポーネントのレコード範囲のみをフェッチします。

最初のファイルごとのプレフィックスの実験では、数十万の小さなファイルが生成され、拒否されました。実装されたルーターは代わりに、グローバルにソートされたエイリアスを制限されたキー範囲にパックします。これにより、1,629 個の検索ファイルが生成され、検索をハッシュベースにすることなく、静的ファイル数の大きなペナルティが回避されました。

ワーカーは、ローカルの Core マニフェストが存在する場合はそれを優先し、フェッチされた検索シャードとレコード シャードをキャッシュし、Core が生成されていない場合は決定論的サンプルにフォールバックするようになりました。ブラウザのチェックでは、`今日` の読み取り値と、`選挙管理委員会` の実際の A/B/C 展開の両方が確認されました。

## 残りの受け入れ作業

これらのアーティファクトを公開する前に、リリース パイプラインで以下を測定する必要があります。

- 最終的なホスティング構成での Brotli および gzip 転送サイズ
- コールドおよびウォームブラウザのルックアップ遅延
- 代表的なクエリ セッション中のブラウザ メモリ
- 短いプレフィックスのキャッシュ動作と総リクエスト数

v3 形式では、シンプルでテスト可能なエンコーディングが意図的に好まれていました。ホスト型測定により、繰り返される POS 文字列がレコード ストレージの重要な部分であることが判明した後、フォーマット v9 では、Sudachi の既存の 16 ビット POS ID と共有圧縮 POS テーブルを使用するようになりました。

フォーマット v4 は、その単純なレイアウトを維持しながら、構造体および A/B ワード ID 配列を 1 バイトの累積コードポイント境界に置き換えます。これにより、コア レコード データは 252,802,733 バイトから 232,461,377 バイトに、フル レコード データは 471,957,614 バイトから 427,577,506 バイトに減少します。さらに重要なことは、表示される結果は、構造および分割コンポーネントの完全なレコードをフェッチしなくなりました。親レコードだけですべてのコンポーネント サーフェスをレンダリングできます。

### ホスティングの結論

測定したファイル数と個別アセットの大きさの範囲では、FullはCloudflare Pagesと互換性があります。Full単独では4,293個、CoreとFullを合わせると6,720個の辞書ファイルを使用します。どちらもFreeプランの上限である20,000ファイルを下回り、生成される最大ファイルも1 MiB未満で、Pagesの個別ファイル上限25 MiBを大きく下回ります。現在の[Cloudflare Pagesの制限](https://developers.cloudflare.com/pages/platform/limits/)も参照してください。

したがって、Pages はデフォルトのホスティング プランのままです。フルの場合は R2 は必要ありません。アップロード期間、多くの辞書バージョンの保持、または展開操作が不便になった場合に、後で最適化するオプションとして残ります。次の決定は、生のコーパス サイズだけではなく、ホストされた圧縮、リクエスト数、ブラウザのメモリの測定に従う必要があります。
