# 正規見出し語フィルタリング

## ステータス

ステージ 1 はブラウザ データ形式 v5 で実装され、固定されたコア エディションとフル エディションの両方で v10 まで保持されます。検索エイリアスとブートストラップ結果は、正規の辞書形式の ID のみを参照します。ニュートラル エクスポートおよびブラウザ レコード シャードには、すべてのソース レコードが引き続き保持されます。ステージ 2 のレコード圧縮は今後のオプションの作業として残ります。

## 製品の特徴

Sudachi のシステム辞書は形態素解析用に構築されています。これには、たとえそれらのレコードが独立した辞書結果として役に立たない場合でも、トークナイザーが必要とする活用状態の辞書レコードが含まれています。

Sudachi Lookup は見出し語ブラウザであり、トークナイザー検査ツールではありません。したがって、その Web データセットは正規の辞書エントリを公開する必要がある一方で、ニュートラル オフライン エクスポートでは固定されたすべての Sudachi レコードが保持され続けます。

固定された Core データはその違いを示しています。次の 14 のレコードの辞書形式として `食べる` が含まれています。

```text
食べ      未然形-一般
食べ      連用形-一般
食べよ    命令形
食べよう  意志推量形
食べりゃ  仮定形-融合
食べる    終止形-一般
食べる    連体形-一般
食べれ    仮定形-一般
食べろ    命令形
食べん    終止形/連体形-撥音便
```

これらのレコードは、Sudachi が実行中のテキストを分析するときに役立ちます。これらを同じ見出し語の個別の検索結果として提示すると、対応する辞書値を追加せずに、ノイズ、別名、候補、およびレコードのハイドレーションが追加されます。

このポリシーは、任意の語形変化した入力を合成または分析しません。 `食べた` は、Sudachi が `食べ` と補助 `た` として扱うため、ルックアップ インデックスの範囲外のままになります。 `た` の正規の補助エントリは、引き続き独立して検索可能です。

## 正規アイデンティティ

フィルタリングでは、`surface == dictionaryForm` などの文字列ヒューリスティックや活用形式のラベルだけではなく、Sudachi の上流の辞書形式の単語 ID を使用する必要があります。

ニュートラル エクスポートでは、すべてのレコードに辞書形式の単語 ID を追加し、それが固定されたシステム辞書内で解決されることを検証する必要があります。ブラウザー データの生成により、各語形変化レコードをその正規エントリにマッピングできます。

ジェネレーターは、同じ表層または辞書形式のスペルを共有する場合でも、明確な正規のアイデンティティを保持する必要があります。同形異義語、読み方、品詞の区別、および完全に独立した語彙エントリは、単にテキストが一致するという理由だけで折りたたまれてはなりません。

アップストリーム レコードに有効な正規 ID がない場合、生成はそれを報告し、デフォルトで保持する必要があります。不確実なヒューリスティックに基づくサイレント削除は受け入れられません。

## 検索動作

`食べる` などの語形変化する見出し語の場合、ブラウザーのインデックスは、その活用状態のレコードではなく正規の結果を公開します。その別名には、標準曲面、辞書形式、正規化形式、および標準読み方が含まれる場合があります。正規化された形式も `食べる` であるバリアント スペルを含め、正規のアイデンティティは別に意図的に保存されます。

予想される動作:

|クエリ |期待される結果 |
| --- | --- |
| `食べる` | `食べる` に一致する正規の見出し語。共役状態の重複はありません |
| `食べ` |プレフィックス検索による正規の `食べる` |
| `タベル` / `たべる` |エイリアスの読み取りによる正規の `食べる` |
| `食べよう` |語形変化した Sudachi レコードであるという理由だけでは一致しません |
| `食べた` |自動活用分析はありません。 `た` は独自のエントリのままです。

リテラルの正規エントリは、正規化された辞書形式の読み別名を引き続き上回る必要があります。一致ソースのメタデータは、正規の見出し語が一致した理由を後で説明する可能性がありますが、フィルタリングはそのメタデータの公開には依存しません。

## 段階的実装

### ステージ 1: フィルター検索エイリアス (v5 ～ v10 で実装)

- ニュートラル エクスポートおよびブラウザー レコード シャード内のすべてのレコードを保存します。
- 上流の辞書形式の単語 ID を追加して検証します。
- 正規のブラウザ エントリに対してのみ検索エイリアスを出力します。
- フィルタリングされたエイリアスからブートストラップ データを再構築します。
- コアおよびフルのエントリ、エイリアス、シャード、候補、およびコールドクエリの削減を測定します。

既存のレコード ID スペースを維持すると、この段階が比較的小さくなり、ユーザーに表示されるポリシーがストレージの圧縮から分離されます。

### ステージ 2: ブラウザー レコードの圧縮

ステージ 1 の動作が受け入れられた後:

- 検索結果で対処できないブラウザーのレコードを除外します。
- コンパクトなブラウザ エントリ ID スペースを割り当てます。
- 検索投稿をそれらの ID に書き換えます。
- ブラウザ ID を固定された Sudachi ワード ID にマッピングするリリース レポートを保持します。

形式 v5 から v10 は v4 の表面境界表現を保持するため、これが容易になります。構造、A、および B 表示データは、コンポーネント レコード ID がブラウザー コーパスに存在し続ける必要がなくなりました。

## 検証と例外

フィルターを有効にする前に、コア生成と完全生成は次のことをレポートして検証する必要があります。

- 合計ソース レコードと正規ブラウザ エントリ。
- 正規 ID ごとのレコードの数と分布。
- 解決できない正規 ID。
- スペルを共有しているものの、明確な区別が残っている正規のエントリ。
- エイリアス、検索シャード、ブラウザー レコードのサイズが変更されます。
- 代表的な動詞、形容詞、助動詞、不規則な形、および非活用見出し語。
- 削除された、またはマッピングされていないブラウザー レコードを参照する検索投稿はありません。

回帰チェックには、少なくとも `食べる`、`する`、`来る`、i 形容詞、na 形容詞、および補助的な `た` が含まれている必要があります。屈折面が独立して語彙化されているエントリは、破棄された屈折レコードではなく、独自の正規アイデンティティを通じて利用可能な状態を維持する必要があります。

Core はエイリアスの 26.4% を削除します。完全に削除すると 14.9% になります。どちらも、レコード ストレージを圧縮せずに約 51 MiB を節約します。ジェネレーターの検証では、すべての非正規レコードが、その表面がレコードの上流の辞書形式と等しい範囲内の正規 ID に解決されることも確認されます。

v5 ブートストラップは、辞書コストだけではなく、実際の 1 文字のクエリに対してランク付けされます。広範なクエリ `い` の場合、最初の 20 個のコア候補は、完全な 51,339 件の結果検索の最初の 20 個の候補と完全に一致します。これにより、「さらに読み込む」後にのみ、より強力な完全一致が表示されるのを防ぎます。

フォーマット v6 は、直接上位 20 位の結果と重複排除された表示レコードを備えたエイリアスベースのブートストラップに取って代わりました。フォーマット v7 は、デコードされたバジェットを 4 MiB に拡張し、転送用に gzip 圧縮し、ブロード エイリアスのしきい値に関係なくプレフィックスを認定するための高いレコード シャード コストを許可します。コアは、9,047 個の適格なプレフィックスから 2,060 個を選択し、39,660 個のレコードを 775,326 転送バイトに埋め込みます。完全選択では、22,501 件中 2,079 件が選択され、40,082 レコードが 784,848 バイトに埋め込まれます。デコードされたサイズは 4 MiB 未満のままです。 `い`、`あい`、`あお`、および `あきの` のコア チェックは、完全な検索の上位 20 を再現し、別の辞書リクエストなしでそれらの結果をレンダリングできます。

フォーマット v8 は、同じ標準フィルターと分割表現を保持しながら、リテラルかな設定を追加します。選択された各かなプレフィックスは、同じデコードされたブートストラップ バジェットの下で、別個のひらがなとカタカナのトップ 20 ランキングを保存できます。コアは、856,126 転送バイトで 3,832 個のブートストラップ キーと 38,144 個の埋め込みレコードを発行します。 Full では、866,314 バイトで 3,841 個のキーと 38,260 個のレコードが発行されます。 `い`/`イ`、`あい`/`アイ`、`あま`/`アマ` などのペアのクエリの場合、入力されたスクリプトに一致するその他の点では等しいサーフェスの結果が最初にランクされます。

フォーマット v9 は、すべてのレコードで繰り返される POS 表示文字列を、元の Sudachi `u16` POS ID と 1,558 個の使用された POS 値の 1 つの共有圧縮テーブルに置き換えます。テーブルは両方のエディションで 10,587 バイトで転送します。コア レコード データは 232,461,377 バイトから 144,421,683 バイトに減少し、その完全な辞書は 88,025,223 バイト (21.1%) 減少しました。完全なレコード データは 427,577,506 バイトから 276,828,309 バイトに減少し、その完全な辞書は 150,713,267 バイト (18.5%) 減少します。 2.5 MiB のデコードされたブートストラップ バジェットにより、ブートストラップと POS を組み合わせた起動転送がコア v8 の 1.7% 以内、フル v8 の 4.1% 以内に抑えられ、v8 よりも多くのプレフィックス キーをカバーします。

## 製品効果

正規フィルタリングにより、結果セットは Sudachi の内部トークナイザ辞書ではなく辞書のように読み取られます。また、エイリアスの数と、ブロード プレフィックスのランキングに必要な作業量も削減されます。サイズとレイテンシの利点は製品ポリシーの結果であり、安全でないグループ化ヒューリスティックを適用する理由ではありません。
