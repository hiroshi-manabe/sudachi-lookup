# 構造一致ルックアップ計画

## ステータス

予定されているオプション機能。このドキュメントは製品とデータの設計を定義しますが、プロジェクトがそれを実装することを約束するものではありません。通常のプレフィックス検索と既存のコンポーネント ナビゲーションの動作は変更されません。

## 目的

Sudachi Lookup は現在、コンポーネントの表面をテキストとして検索することで、化合物からその表示されているコンポーネントの 1 つに移動します。構造一致ルックアップは、逆の探索をサポートします。

> 直接の `word_structure` が次で始まるか終わる正規の見出し語を検索します。
> このエントリ。

たとえば、構造が `選挙 / 管理 / 委員会` の場合、`委員会` からの最終位置ルックアップは `選挙管理委員会` を返すことがあります。 `自` からの最初の位置のルックアップは、その構造が `自 / 文化` の場合、`自` の通常の接頭辞一致に混合することなく、`自文化` を分離する可能性があります。

これは関係クエリであり、プレフィックスまたはサフィックス文字列の一致ではありません。 `委員会` の入力されたクエリは、テキストによるエイリアスの検索を続行します。関係モードは特定の辞書結果からのみ入るため、Sudachi の単語のアイデンティティが保持されます。

## 初期セマンティック スコープ

最初の実装では、意図的に狭いセマンティクスを使用する必要があります。

- `word_structure` の最初と最後の直接コンポーネントを 2 つの明示的な位置としてサポートします。
- A または B ユニットの分割リストを検査しないでください。
- ネストされた構造コンポーネントに再帰的に降下しないでください。
- 一致するサーフェスのプレフィックスまたはサフィックスから関係を推測しないでください。
- 既存の見出し語フィルタリング ポリシーに従って、正規の辞書形式の親エントリのみを返します。
- 関係を追加する前に、辞書形式の単語 ID を使用して参照先コンポーネントを正規化することで、検索可能な見出し語で内部の屈折状態の ID に由来する関係を扱うことができます。
- コンポーネント ID の下で親の重複を排除します。

内部構造位置のインデックス付けは、後の拡張として考えられますが、これらのセマンティクスを黙って拡張するのではなく、別の関係モードである必要があります。

## インタラクション モデル

### 関係モードに入る

結果は、コンパクトでグループ化された 1 つの二次アクションを提供します。

```text
構造一致: 先頭 / 末尾
```

`先頭` と `末尾` は、結果の標準単語 ID を使用し、対応する位置を直接入力する別のリンクです。アクセス可能な完全な名前は、表示されるラベルよりも意図的により明確になっています。

- `「自」を構造の先頭に持つ語を検索`
- `「委員会」を構造の末尾に持つ語を検索`

どちらのリンクもテキスト検索を実行したり、結果の分割展開を切り替えたりすることはありません。グループ化されたアクションは視覚的に二次的なものにし、実装前に既存の見出し語、Web 検索、メタデータ、分割コントロールに対してテストする必要があります。

### 検索コントロール

通常モードでは、編集可能なテキスト入力が保持されます。

```text
Sudachi辞書を検索
[ 委員会                                      ]
```

構造一致モードでは、編集可能な値が 1 つの編集不可能なトークンと 2 つの位置のセレクターに置き換えられます。

```text
構造一致
[  委員会  ×  ]   [ 先頭 | 末尾 ]
```

トークンは、単にその目に見える表面だけではなく、選択された単語 ID を表します。その表面はそのアイデンティティのラベルです。この区別により、同形異義語は明確に保たれます。同じ文字を入力すると通常のテキスト検索が実行され、結果を入力すると同一性ベースの関係検索が実行されます。

検索コントロールは次のように動作する必要があります。

- トークンにはテキスト キャレットがないため、その場で編集できません。
- `×` ボタンは関係モードをクリアし、空の通常の入力に戻り、その入力に焦点を当てます。
- クリア ボタンには、`構造検索を解除して通常検索に戻る` などのアクセス可能な名前が付いています。
- 検索コントロールがフォーカスされているときに、削除または Backspace キーを押すとトークンがクリアされます。
- トークンをクリアすると、未処理の関係作業がキャンセルまたは無効になります。
- ブラウザーの「戻る」と「進む」により、モードと選択した ID の両方が復元されます。
- IME 合成は通常のテキスト モードにのみ適用されます。
- 直接 URL は、選択されたレコードがその表面と読み取り値を提供するまで、最初はトークン読み込み状態を示します。
- `先頭` と `末尾` を切り替えると、選択された ID が保持され、すぐに他の関係の検索が開始されます。
- 位置スイッチは、トグルごとにバック ステップを作成するのではなく、現在の関係履歴エントリを置き換えます。

検索コントロールが既に関係を表現しているため、表示される結果の見出しはコンパクトなままです。

```text
検索結果
```

アクセシブルなステータスの説明には、`「自」を構造の先頭に持つ語を検索中` などの完全な文言を使用できます。空、読み込み中、エラー、結果数、および継続メッセージは、プレフィックス テキスト検索が実行中であることを暗示してはなりません。

### URL と状態

相互に排他的なクエリ表現を使用します。

```text
?q=委員会                         ordinary text lookup
?structure=first&component=123456   first-position Structure Match
?structure=last&component=123456    last-position Structure Match
```

構造一致モードは、`structure` が `first` または `last` で、`component` がアクティブな辞書アーティファクトの有効な符号なし単語 ID である場合にのみ有効です。無効な位置または利用できない ID がある場合は、回復可能な日本語エラーが生成され、通常の検索に戻る必要があります。 URL はサーフェスを複製する必要はありません。直接アクセスすると、ID の既存のレコード シャードを読み込むことができますが、同じセッションの履歴では、キャッシュされたラベルがすぐにレンダリングされる可能性があります。

UI 状態は、並列クエリ フラグではなく、タグ付きユニオンとしてモデル化する必要があります。

```ts
type LookupMode =
  | { kind: "text"; query: string }
  | {
      kind: "structure-match";
      position: "first" | "last";
      componentId: number;
      surface?: string;
    };
```

## 逆索引データ

### 論理形式

構造の単語 ID 参照が表示境界に縮小される前に、ニュートラル エクスポートからオフラインでインデックスを構築します。

```text
canonical component word ID
    -> first-position pre-ranked canonical parent word IDs
    -> last-position pre-ranked canonical parent word IDs
```

このインデックスには、コンポーネント文字列、親文字列、POS 値、または完全なレコードは属しません。既存のレコード シャードは、逆インデックスによって ID が提供された後、表示されている親をハイドレートします。

親の投稿はオフラインで決定的にランク付けされる必要があるため、広範な関係では最初のページを表示する前に何千もの親レコードをロードする必要がありません。最初のランキングでは、可能な限り通常の結果の原則を再利用する必要があります。

1. Sudachi のワードコストが低い。
2. 親サーフェスが短くなります。
3. より低い親単語 ID を安定した最終タイブレーカーとして使用します。

正確なランキングは、フォーマットが凍結される前に、一般的で非常に広範なコンポーネントについて測定する必要があります。

### エンコーディングとシャーディング

コンポーネントの単語 ID 範囲を遅延逆インデックス シャードにルーティングする、バージョン管理されたマニフェスト セクションを追加します。候補バイナリ シャードには次のものが含まれます。

- フォーマットマジックとバージョン。
- デルタエンコードされたコンポーネント単語 ID。
- コンポーネントごとの最初と最後の位置の投稿数。
- 事前にランク付けされた親 ID は、コンパクトな符号なし値または符号付きデルタとしてエンコードされます。
- 大規模な投稿で必要な場合は、オプションのリストごとの継続オフセット。

現在のブートストラップや POS テーブルと同様に、各シャードを gzip で圧縮し、Web Worker 内でデコードします。シャードあたりの転送量はおよそ 64 ～ 128 KiB を目標とし、明示的な最大値を既存のページ資産予算よりも低く抑えます。コンポーネント ID からシャードを選択するには、マニフェスト ルーティング テーブルで十分である必要があります。通常の起動時や通常のテキスト検索時には、逆引きインデックスはフェッチされません。

### 測定スケール

2026 年 7 月のニュートラル コアおよびフル エクスポートでは、正規の親フィルタリング後の最初と最後の合計カウントが次のようになります。

|測定 |コア |フル |
| --- | ---: | ---: |
|正規の検索可能なエントリ | 1,198,652 | 2,452,463 |
|構造を持つ正規の親 | 573,277 | 1,487,960 |
|最初と最後の位置の関係 | 1,146,554 | 2,975,920 |
|明確な直接の最初のコンポーネント | 44,140 | 149,906 |
|明確な直接最後のコンポーネント | 114,943 | 290,200 |
|どちらの位置でも異なるコンポーネント | 146,722 | 395,783 |

コンポーネント ID デルタ、親 ID デルタ、および生成された 1 つの gzip ストリームを使用したシミュレーション:

|アーティファクト |コア |フル |
| --- | ---: | ---: |
|圧縮されたデコードされたバイト | 2,479,863 | 6,658,639 |
|シミュレートされた gzip バイト | 1,486,728 | 4,461,778 |
|本番環境の共有予算 | 1.7–2.1 MiB | 4.8–5.8 MiB |

シングルストリーム gzip の結果は下限実験であり、コミットされたファイル形式ではありません。独立したシャード、単調に並べ替えられた親 ID、ヘッダー、およびルーティング メタデータではなく事前にランク付けされた場合、この値が増加する可能性があります。したがって、形式のプロトタイプでは、実際のエンコーディングが測定されるまで、8 MiB という保守的な完全転送バジェットを適用する必要があります。

コンテキストとして、すべての個別のコンポーネント位置にインデックスを付けると、同じシミュレーションでコアの圧縮で約 1.40 MiB、フルの圧縮で 4.63 MiB が生成されました。したがって、ストレージは主要な製品リスクではありません。

## Worker と結果の読み込み

テキスト検索メッセージをオーバーロードするのではなく、明示的な関係要求を使用してワーカー プロトコルを拡張します。流れは次のとおりです。

```text
component word ID and first/last position
    -> manifest range route
    -> one lazy reverse-index shard
    -> stable first page of pre-ranked parent IDs
    -> existing result slots and record-shard hydration
    -> continuation pages from the same posting list
```

通常の結果スロット メカニズムを再利用できます。リレーションシップ セッションでは、依然として単調に増加するリクエスト ID、古い応答の拒否、安定した重複排除、および漸進的なレコード ハイドレーションが必要です。ブートストラップとテキスト検索シャードは関係しません。

20 個の初期 ID を返し、次に既存の 50 個の結果の継続ページを返します。非常に広範な投稿には、広範なテキストの結果と同じ自動継続動作と明示的な再試行パスが必要です。デコードされたリバース シャードをワーカーのメモリ バジェットの下で個別にキャッシュします。

## 検証

ビルダーと生成データ検証ツールは、次のことを証明する必要があります。

- すべてのインデックス付きコンポーネントと親 ID が範囲内にあります。
- すべての親は正規の検索可能なアイデンティティです。
- すべての関係は、ニュートラル エクスポート内の正規化された最初または最後の直接 `word_structure` 参照に対応します。
- 1 つの投稿リスト内で親が重複することはありません。
- すべての投稿リストは、宣言された決定的なランクに従います。
- シャード範囲は順序付けされており、重複がなく、完全で、サイズが制限されています。
- コア数とフル数、およびバイト合計が再現可能にレポートされます。
- サンプル フィクスチャは、同形異義語、親のないコンポーネント、複数の親、広範なコンポーネント、および辞書形式 ID による正規化をカバーしています。

アプリケーション テストは以下をカバーする必要があります。

- テキストだけからではなく、特定の結果からモードに入る。
- コンパクトな `構造一致: 先頭 / 末尾` リンクは、結果を切り替えることなく、正しい ID と位置を選択します。
- 編集不可能なトークンの動作とクリアボタンのラベル。
- `先頭` と `末尾` を切り替えると、ID が保持され、URL が更新され、新しく選択された関係リクエストのみが開始されます。
- 削除とバックスペースでトークンをクリアします。
- 通常の入力は通常のテキスト検索のままです。
- URL の直接ロードとバック/フォワード復元。
- 古いテキストや関係の応答がアクティブ モードを置き換えることはありません。
- 日本語の空、エラー、ハイドレーション、継続、および再試行の状態。
- 結果アクションも分割展開を切り替えません。
- キーボード、スクリーン リーダー、狭い画面、および長いラベルの動作。

## 実装フェーズ

2026 年 7 月 20 日にブラウザデータ形式 v10 で実装されました。生成されたインデックスは、正規のコンポーネント ID と親 ID、オフライン親ランキング、gzip 圧縮、およびデコードされた約 128 KiB 範囲のシャードを使用します。マニフェストにも起動ブートストラップにも投稿データが埋め込まれていないため、完全に怠惰なままです。

測定されたリリース出力:

|測定 |コア 20260428 v10 |完全な 20260428 v10 |
| --- | ---: | ---: |
|正規コンポーネント | 145,242 | 393,411 |
|最初の関係 | 573,277 | 1,487,960 |
|最後の関係 | 573,277 | 1,487,960 |
|シャード | 50 | 128 |
|デコードされたバイト | 6,329,620 | 16,625,892 |
|圧縮バイト | 3,293,246 | 8,875,874 |
|最大の圧縮シャード | 82,340 B | 83,790 B |

### フェーズ 1: フォーマット実験 (完了)

- v9 の出力を変更せずに、測定コードを辞書ビルダーに追加します。
- ソース ID の順序、絶対バリアント、および事前にランク付けされた符号付きデルタ エンコーディングを比較します。
- シャードされたコアバイトとフルバイト、最大シャードサイズ、一般的な投稿分布を測定します。
- コンポーネント参照の正規化動作を確認します。
- 新しいブラウザ データ フォーマット バージョンは、これらの結果が 8 MiB のフル バジェットを超えた後にのみ凍結されます。

### フェーズ 2: 生成されたデータと検証 (完了)

- 逆インデックスのシャードを生成し、ルーティング メタデータをマニフェストします。
- 関係のケースを使用して決定論的なサンプル フィクスチャを拡張します。
- コアとフルの構造検証とランキング検証を追加します。
- すべてのリバース アセットを遅延させ、ブートストラップ スタートアップ トラフィックから除外します。

### フェーズ3: Workerの構造一致セッション（完了）

- 個別のリクエストとセッション タイプを追加します。
- 逆シャードをルーティングしてキャッシュします。
- 安定した結果スロットを再利用し、ハイドレーション、ページネーション、再試行、古い作業のキャンセルを記録します。
- 表示されるインターフェースを変更する前に、ユニットと統合フィクスチャを追加してください。

### フェーズ 4: トークンモード インターフェイス (完了)

- 完全にアクセス可能なラベルを使用して、グループ化された結果レベルの `構造一致: 先頭 / 末尾` アクションを追加します。
- リレーションシップ モードでは、編集可能なテキストを編集不可能な ID トークンに置き換えます。
- 一致する `先頭 / 末尾` セレクターとコンパクトな `構造一致` モード ラベルを検索コントロールに追加します。
- クリア、削除、バックスペース、フォーカス、URL、履歴、および直接読み込みの動作を実装します。
- 関係固有の日本語ステータスと結果のコピーを追加します。
- 通常の入力や IME の動作を弱めることなく、応答性が高くアクセスしやすい操作を検証します。

### フェーズ 5: 段階的リリース (進行中)

- サンプルをローカルで、およびページ プレビューを通じて実行します。
- コールド リバース シャードおよびレコード シャード リクエストを含むコア ステージングをデプロイおよび測定します。
- コアの動作と予算が許容できる場合にのみ、完全なステージングをデプロイします。
- 有用性を観察し、品質をランク付けした後、本番環境の有効化を別個の決定として扱います。

## 延期された選択肢

測定やユースケースが正当化されない限り、最初の実装にはこれらを含めるべきではありません。

- 内部構造位置でコンポーネントを照合します。
- 再帰的な祖先の走査。
- A または B 分割の逆関係。
- 複数のコンポーネントトークンを組み合わせる。
- 編集可能な関係トークン。
- 関係の親を通常のプレフィックス結果に混合します。
- サーバー側の分析、ランキング、または検索。
