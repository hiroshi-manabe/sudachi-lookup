# 複合ナビゲーション対話仕様

## ステータス

ブラウザデータ形式v4で実装されています。構造および A/B ワード ID 参照は、[Compact Split Boundary Format](split-boundary-format.md) で定義されているように、オフラインで 1 バイト境界に解決されます。インタラクション、履歴、拡張、およびアクセシビリティのモデルは、v3 プロトタイプから変更されていません。

## 目的

検索結果では、次の 2 つの関連するアイデアが混同されることなく表示される必要があります。

1. エントリの辞書構造は、さらに検索するための便利な宛先を提供します。
2. Sudachi の A/B/C モードは、セグメンテーションの粒度全体でエントリがどのように変化するかを説明します。

折りたたまれた結果はナビゲーションを優先します。展開された結果はセグメンテーション モードを説明します。表示されているすべてのコンポーネントが次のクエリになるため、手動でコピーする必要がなく、複合探索が継続的に行われるようになります。

## エントリーユニットラベル

すべての結果は、コンパクトなユニットバッジから始まります。

- エントリに A ユニット分割がない場合は、`[A]`。
- `[B]` A ユニット スプリットはあるが B ユニット スプリットがない場合。
- B ユニット分割がある場合は `[C]`。

ジェネレーターはこの分類を明示的に保存するか、分割メタデータから派生してテストする必要があります。インターフェイスは、表面のテキストからそれを推測してはなりません。

## 折りたたまれたプレゼンテーション

### A エントリ

A エントリーはすでに最短単位です。その表面は直接表示され、折りたたまれた見出しでは分解されません。現在の正規化されたクエリと異なる場合、サーフェス全体がルックアップ アクションになります。これにより、`権` などの広範なクエリで `権利` などの A 結果に移動できるようになりますが、同じクエリを繰り返しても効果がないため、正確な `権` の結果はプレーン テキストのままになります。

比較では、NFKC 正規化、日本語ロケールの小文字化、およびトリミングが使用されます。かなと漢字は同等のものとして扱われないため、`けん` などの読み取りクエリは、表示されたサーフェイス `権` にナビゲートできます。

正規化されたメタデータ値は同じルールに従います。つまり、現在の正規化されたクエリおよび no-op 自己一致のプレーン テキストと異なる場合は、検索アクションになります。これをアクティブ化しても、結果カードの分割展開を切り替えてはなりません。メタデータのラベルと値は明示的なグリッド列を占有するため、隣接する POS 行と正規化された行を一緒に実行することはできません。

すべての結果では、見出し語のすぐ下に、読みの横に 2 番目の `ウェブで検索` アクションも表示されます。これにより、クリック可能な構造コンポーネントにアクションを混ぜることなく、エントリーに関連付けられたアクションが保持されます。新しいタブで Google の結果画面を検索し、分割展開を切り替えてはなりません。表示ラベルはプロバイダー中立のままであるため、将来のバージョンでは結果のレイアウトを変更せずにプロバイダーを構成できるようになります。プロバイダーの選択は、将来の可能性のあるタスクにすぎず、最初のリリースの要件ではありません。外部リクエストは、ユーザーがこのアクションをアクティブ化した後にのみ行われます。

```text
[A]  選挙
     センキョ
```

### B および C エントリ

B または C エントリは、`word_structure` コンポーネントを使用して表示されます。各コンポーネントは独立した検索アクションです。

```text
[C]  [選挙] [管理] [委員会]
     センキョカンリイインカイ
```

上の括弧は、個別の対話型ターゲットを示しています。完成したビジュアルデザインでは、すべてのコンポーネントを文字通りの角括弧で囲む必要はありません。

構造は別の分割モードではありません。これはメインエントリのナビゲート可能なプレゼンテーションを制御し、A/B/C 行はセグメンテーション動作を説明します。

データ形式では、2 つまたは 3 つを超える構造コンポーネントが許可されます。コンポーネントは狭い画面上で自然にラップする必要があり、固定カーディナリティに依存してはなりません。 Structure が空であるか解決できない場合、結果はエントリ自体の表面に戻ります。

## 展開動作

結果の非コンポーネント領域をアクティブにすると、その分割の詳細が切り替わります。展開された行には、結果自体の単位より細かいモードのみが表示されます。

|結果ユニット |展開された行 |
| --- | --- |
|あ |なし |
| B |のみ |
| C | B、次に A |

たとえば、C エントリを展開すると次が生成されます。

```text
[C]  [選挙] [管理] [委員会]
     センキョカンリイインカイ

[B]  [選挙] [管理] [委員会]
[A]  [選挙] [管理] [委員] [会]
```

B エントリを展開すると、以下が生成されます。

```text
[B]  [委員] [会]
     イインカイ

[A]  [委員] [会]
```

展開された B 行または A 行のすべてのコンポーネントは、折りたたまれた見出しの Structure コンポーネントと同様に、独立してナビゲート可能です。

## コンポーネント ナビゲーション

構造、A、または B コンポーネントをアクティブにすると、新しいルックアップが実行されます。

1. コンポーネントの表示されたサーフェスを新しいクエリとして使用します。
2. 検索入力内のテキストを置き換えます。
3. 前の結果の展開状態を閉じます。
4. 新しいクエリの結果をロードして表示します。
5. URL を新しいクエリ (`?q=委員会` など) に移動します。

コンポーネント ナビゲーションではブラウザ履歴エントリを作成し、[戻る] で前の複合ルックアップに戻り、そのクエリを復元する必要があります。通常の 1 文字ずつの入力では、キーストロークごとに 1 つの履歴エントリが作成されることを避ける必要があります。通常の検索デバウンス後に現在のクエリ URL を置き換えることができます。

表面による検索は意図的です。コンポーネント サーフェスは複数の辞書エントリに対応する場合があり、次の結果画面では、参照された単語 ID のみを黙って選択するのではなく、それらの同形異義語を公開する必要があります。

別個の将来の関係モードは、特定の結果アイデンティティから開始し、直接の構造がそのエントリで始まるか終わる正規の親を見つけることができます。このサーフェスベースのナビゲーション動作を変更してはなりません。そのコンパクトな `構造一致: 先頭 / 末尾` アクション、トークン スタイルの検索制御、データ、URL 状態、および段階的な実装は、[構造一致検索プラン](structure-match-lookup.md) で指定されています。

## イベントとマークアップ モデル

現在のプロトタイプでは、結果カード全体が 1 つのボタンになります。ネストされた対話型コントロールは無効であり、混乱を招くキーボード動作を引き起こすため、そのモデルにはコンポーネントのボタンやリンクを含めることはできません。

実装された結果は、代わりに次のようなボタン以外のコンテナーになる必要があります。

- セマンティックな結果の見出しとメタデータ。
- コンポーネントごとに個別のボタンまたはクエリ リンクを使用します。
- `aria-expanded` および `aria-controls` を使用した専用の拡張コントロール。
- 同じ展開アクションを呼び出す、オプションのバックグラウンド クリック ターゲット。

コンポーネントのアクティブ化は、バックグラウンドの展開アクションに伝播してはなりません。拡張コントロールには、表示可能なキーボード フォーカス状態と、`Show split modes for 選挙管理委員会` などのアクセス可能な名前が必要です。

ユニットバッジは有用な区別を視覚的に伝えますが、その A/B/C テキストは支援技術で利用できる状態にしておく必要があります。色だけでユニットをエンコードしてはなりません。

## データ要件

抽出データセットは、`word_structure`、A-split、および B-split の単語 ID 参照を保持します。ブラウザーのビルドは、それらを親サーフェス内の累積コードポイント境界に解決します。ビルド パイプラインは次のことを行う必要があります。

- 検証と来歴のために上流の単語 ID を抽出レポートに保存しますが、ブラウザーの記録からは省略します。
- 参照されているすべてのシステム ワードが範囲内にあることを検証します。
- 参照されるコンポーネントの長さが、親サーフェスをカバーする 1 バイト境界を厳密に増加させて形成することを検証します。
- 構造、A、B の境界を親レコードに積極的に保存します。
- コンポーネント レコード シャードをロードせずに、親サーフェスからコンポーネント ラベルを再構築します。
- レポート構造のエントリ数、参照数、未解決の参照数、およびコアとフルの両方のコンポーネント数の分布。

構造フィールドは、そのコンポーネントがエントリの A または B 分割に偶然一致した場合でも、論理モデルとバイナリ モデルで区別されたままでなければなりません。

## ランキングと状態への影響

ナビゲーションによって検索ランキングが変わることはありません。クリックされたコンポーネントは通常のクエリになり、入力されたものと同じルールによってランク付けされます。

展開された状態は、現在のクエリ結果セットに属します。コンポーネントのナビゲーションなど、クエリが変更されるとクリアされます。後の機能強化では、「戻る」に移動するときに展開されたエントリが復元される可能性がありますが、最初の実装では、クエリとスクロールの復元が展開復元よりも重要です。

## 受け入れ基準

インタラクションは自動化され、ブラウザー テストで次のことが示されると完了します。

- A、B、C のエントリーには正しいバッジが与えられます。
- エントリには意味のない分割拡張は提供されません。
- A サーフェスは、現在の正規化されたクエリと異なる場合にルックアップを開始できますが、正確な自己一致はプレーン テキストのままです。
- 正規化された値が異なると、カードを切り替えることなく検索が開始されますが、完全な自己一致はプレーン テキストのままです。
- B エントリは A のみに展開されます。
- C エントリは B と A のみに展開されます。
- B および C の折りたたまれた見出しは、利用可能な場合は構造コンポーネントを使用します。
- すべての構造、A、および B コンポーネントはルックアップを開始できます。
- コンポーネントをアクティブ化しても、元の結果は切り替わりません。
- Web 検索では、結果を切り替えることなく、新しいタブで結果画面が開きます。
- ブラウザー戻るは、コンポーネントのナビゲーション後に前のクエリに戻ります。
- ホモグラフは、コンポーネント サーフェスを検索した後も表示されたままになります。
- キーボードとスクリーン リーダーのユーザーは、ナビゲーションと拡張を区別できます。
- モバイル画面では、長いコンポーネント シーケンスが水平方向にオーバーフローすることなく折り返されます。

この機能が公開される前に、通常の TypeScript、アプリケーション ビルド、サンプル形式のテスト、Rust テスト、および生成されたコア/完全な構造検証に合格する必要があります。
